"이슈(Issue)에서 코드가 시작된다"는 말이 추상적으로 들릴 수 있죠. 개발자이자 이 시스템을 직접 세팅해야 하는 구현자 입장에서, 깃허브 코파일럿(특히 Copilot Workspace)이 어떻게 스펙 문서를 집어삼키고 Plan과 Tasks를 뱉어내는지 아주 구체적인 워크플로우를 해부해 드릴게요.

커서(Cursor)나 클로드(Claude)를 쓰면 개발자가 일일이 창을 띄워놓고 "이 이슈 내용 복사할 테니까 읽어봐. 그리고 이 폴더에 파일 만들어"라고 수동으로 '컨텍스트 배달'을 해야 합니다. 반면, 깃허브 코파일럿 워크플로우에서는 이 과정이 완전히 자동화된 파이프라인으로 연결됩니다.

구현자 관점에서 세팅하고 실행하는 4단계 실전 프로세스입니다.

1단계: 스펙킷(Spec Kit) 양식의 Issue Template 강제화
모든 것은 GitHub Issue에서 시작됩니다. 기획자나 PM이 아무렇게나 요구사항을 적지 못하게, .github/ISSUE_TEMPLATE/ 폴더에 sdd-feature.yml 같은 템플릿을 세팅해 둡니다.

구현 포인트: 이슈 템플릿 자체가 스펙킷의 Specify(요구사항 및 제약조건) 항목을 채우도록 강제합니다.

Jira 연동(선택): 조직에서 Jira를 메인으로 쓴다면, Jira에서 User Story가 확정될 때 웹훅(Webhook)을 통해 GitHub Issue로 해당 내용이 스펙 양식에 맞춰 자동 생성되도록 파이프라인을 뚫어줍니다. 이렇게 하면 **'이슈 = 단일 진실 공급원(SSOT)'**이 완성됩니다.

2단계: "Open in Copilot Workspace" (에이전트 호출)
개발자가 해당 GitHub Issue에 들어갑니다. 우측 상단이나 코멘트 창에 있는 Open in Workspace 버튼을 클릭합니다.

구현 포인트: 이 버튼을 누르는 순간, 코파일럿 에이전트는 해당 이슈의 텍스트(Specify)뿐만 아니라, 현재 연결된 리포지토리의 전체 코드베이스, 디렉토리 구조, 그리고 .github/copilot-instructions.md (우리가 앞서 만든 행동 강령)까지 단숨에 컨텍스트로 빨아들입니다.

개발자가 로컬 IDE에서 파일 트리 열어가며 AI에게 컨텍스트를 먹여줄 필요가 0(Zero)이 됩니다.

3단계: AI의 명세서 번역 (Plan & Tasks 자동 생성)
Workspace 화면이 열리면, 코파일럿은 코드를 바로 짜지 않습니다. 이슈 내용을 분석하여 **'Proposed Plan(제안된 계획)'**을 마크다운 형태로 화면에 뿌려줍니다.

구현 포인트 (가장 중요): AI가 내맘대로 계획을 짜지 못하게 통제해야 합니다. 프로젝트 루트의 .github/copilot-instructions.md에 다음 룰을 박아두면 됩니다.

"Copilot Workspace에서 Plan을 생성할 때, 반드시 Spec Kit 프레임워크의 Plan.md(아키텍처 및 변경 파일 목록)와 Tasks.md(체크리스트 형태의 작업 단위) 양식을 엄격하게 준수하여 출력할 것."

결과: AI는 스스로 "1. src/api/payment.ts 파일 생성 (Plan), 2. 결제 검증 로직 작성 (Task 1), 3. DB 스키마 업데이트 (Task 2)" 식으로 작업 명세서를 분할해 냅니다.

4단계: Human-in-the-loop (개발자의 리뷰 및 구현 승인)
코파일럿이 뽑아낸 Plan과 Tasks를 개발자가 화면에서 바로 리뷰합니다.

"어? 이 API 엔드포인트는 레거시 인터페이스랑 안 맞는데?" 싶으면 그 자리에서 채팅으로 "Task 2번, 레거시 ACL 어댑터 통하도록 수정해"라고 지시합니다.

계획이 완벽해지면 Implement(구현) 버튼을 누릅니다.

코파일럿이 Tasks 리스트를 하나씩 체크해가며 실제 코드를 변경하고, 이 모든 변경 사항을 묶어서 **자동으로 하나의 Pull Request(PR)**로 올려줍니다. PR 본문에는 아까 확정된 Plan.md와 Tasks.md가 깔끔하게 첨부됩니다.
