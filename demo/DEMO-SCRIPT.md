# 🎬 SDD 워크플로우 라이브 데모 스크립트

> 전체 소요 시간: ~12분
> 대상: 개발팀
> 준비물: GitHub 리포 (이 프로젝트가 push된 상태), Copilot Workspace 접근 권한
> 참고: 이 프로젝트는 백엔드(FastAPI, http://localhost:8000)와 프론트엔드(React+Vite, http://localhost:5173)로 구성

---

## 🎙️ 인트로 (1분)

> **핵심 메시지**: "이슈 하나로 코드까지 — 수동 컨텍스트 배달의 종말"

**말할 것:**
- "커서나 클로드를 쓸 때, 우리가 하는 일의 절반은 '컨텍스트 배달'입니다. 이슈 내용 복사, 파일 트리 열기, '이 파일 읽어봐' 지시..."
- "GitHub Copilot의 SDD 워크플로우에서는 **이슈 = 단일 진실 공급원(SSOT)**이 되고, 컨텍스트 배달이 완전히 자동화됩니다."
- "오늘 이 4단계 파이프라인을 라이브로 보여드리겠습니다."

## 1단계: 스펙킷 양식의 Issue Template (2분)

> **핵심 포인트**: 기획자/PM이 아무렇게나 요구사항을 쓰지 못하게 **양식을 강제**한다

### 시연 동작:
1. GitHub 리포 → **Settings > Features > Issues** 확인 (이슈 기능 활성화)
2. **`.github/ISSUE_TEMPLATE/sdd-feature.yml`** 파일을 보여준다
3. GitHub UI에서 **"New Issue"** 클릭 → "SDD Feature Request" 템플릿이 뜨는 것을 보여준다
4. `demo/sample-issue.md` 내용을 복사하여 양식에 붙여넣고 이슈를 생성한다

### 강조할 것:
- "기능 요약, 사용자 스토리, 수용 기준, 기술 제약 — 이 4가지가 **필수 입력**입니다."
- "이렇게 구조화된 이슈가 있으면, 사람이든 AI든 '다음에 뭘 해야 하는지' 바로 파악할 수 있습니다."
- "이게 스펙킷(Spec Kit) 프레임워크의 **Specify** 단계입니다."

---

## 2단계: "Open in Copilot Workspace" (2분)

> **핵심 포인트**: 버튼 하나로 **이슈 텍스트 + 전체 코드베이스 + copilot-instructions.md**가 자동으로 컨텍스트에 로딩

### 시연 동작:
1. 방금 생성한 Issue 페이지에서 **"Open in Copilot Workspace"** 버튼을 클릭
2. Workspace가 로딩되는 동안 잠시 대기

### 강조할 것:
- "지금 이 순간, Copilot은 3가지를 동시에 읽고 있습니다:"
  - ① 이슈 텍스트 (요구사항)
  - ② 리포의 전체 코드 구조 (`app/models/`, `app/routers/` 등)
  - ③ `.github/copilot-instructions.md` (우리가 정한 행동 강령)
- "커서에서 이걸 하려면? 파일 5개 열고, 이슈 내용 복사하고, '이 규칙 지켜' 프롬프트 붙이고... 여기서는 **버튼 한 번**입니다."

---

## 3단계: AI의 Plan & Tasks 자동 생성 (3분)

> **핵심 포인트**: AI가 코드를 바로 짜지 않고, **먼저 계획을 세운다** + 우리 양식을 따른다

### 시연 동작:
1. Copilot Workspace 화면에서 **"Proposed Plan"** 섹션을 보여준다
2. Plan이 우리가 정한 양식(아키텍처 변경 요약 + 변경 파일 목록 테이블)을 따르는지 확인
3. Tasks 체크리스트가 파일 단위로 분할되었는지 확인

### 강조할 것:
- "이 Plan의 양식을 보세요. '아키텍처 변경 요약'과 '변경 파일 목록 테이블' — 이건 AI가 알아서 한 게 아니라, **copilot-instructions.md에 우리가 강제한 양식**입니다."
- "Tasks도 보세요. 파일별로 나뉘어 있고, 마지막에 검증 항목이 있습니다. 역시 우리 규칙입니다."
- "`app/models/todo.py`에 `due_date`, `priority` 추가, `app/data/store.py` 반영, 그리고 프론트엔드 `frontend/src/types.ts`, `TodoForm.tsx`, `TodoItem.tsx` 변경... 이슈의 '영향 범위'와 일치하는지 비교해 보세요."

### (만약 양식을 안 따랐다면):
- "양식을 건너뛴 부분이 있네요. 이럴 때 **바로 다음 단계에서 수정 지시**를 내릴 수 있습니다."

---

## 4단계: Human-in-the-loop + Implement + PR (3분)

> **핵심 포인트**: 개발자가 **리뷰하고 수정 지시한 뒤에야** 코드가 생성된다 (맹목적 자동화 아님)

### 시연 동작:
1. Plan/Tasks를 검토한 뒤, 필요하면 채팅으로 수정 지시
   - 예: "Task 2에서 priority의 기본값이 'medium'인지 확인해 줘"
   - 또는: "하위 호환성 테스트 Task를 추가해 줘"
2. 확정 후 **"Implement"** 버튼 클릭
3. Copilot이 Tasks를 하나씩 체크하며 코드를 변경하는 과정을 보여준다
4. 변경 완료 → **자동 PR 생성** 확인
5. PR 본문에 Plan/Tasks가 첨부된 것을 보여준다

### 강조할 것:
- "AI가 바로 코드를 짜는 게 아닙니다. 개발자가 **계획을 승인한 뒤에야** 구현이 시작됩니다."
- "그리고 변경 사항이 하나의 PR로 묶이고, PR 본문에 '왜 이렇게 바꿨는지' Plan이 그대로 들어갑니다."
- "코드 리뷰어도 PR만 보면 '이슈 → 계획 → 코드'의 전체 맥락을 파악할 수 있습니다."

## 🏁 마무리 (1분)

> **핵심 메시지**: 3가지 키워드

**말할 것:**
1. **SSOT** — "이슈가 유일한 진실의 원천입니다. 기획서 따로, 슬랙 따로, 코드 따로... 가 아닙니다."
2. **자동화** — "컨텍스트 배달, 계획 수립, 코드 생성, PR 작성이 하나의 파이프라인으로 연결됩니다."
3. **품질 게이트** — "Template이 입력 품질을, instructions.md가 출력 양식을, Human review가 최종 판단을 보장합니다."

> "이게 Spec-Driven Development입니다. 질문 있으시면 받겠습니다."

## ⚡ 트러블슈팅 치트시트

| 상황 | 대응 |
|------|------|
| Copilot Workspace 로딩이 느림 | "코드베이스를 분석하는 중입니다" 설명 + 잠시 대기 |
| Plan 양식이 instructions.md를 안 따름 | "이런 경우 채팅으로 '양식 따라줘'라고 지시하면 됩니다" — 라이브로 수정 시연 |
| Implement 중 에러 발생 | "실제 개발에서도 이런 일이 있죠. AI가 완벽하진 않습니다. 하지만 80%를 자동화하고 20%만 수정하면 됩니다" |
| "Open in Workspace" 버튼이 안 보임 | Copilot 라이선스 / Workspace 기능 활성화 확인 필요 |
| 프론트엔드가 안 뜸 | `cd frontend && npm run dev` 확인, 백엔드 CORS 설정 확인 |
